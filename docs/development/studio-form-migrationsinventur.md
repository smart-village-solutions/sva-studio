## Priorisierte Migrationsreihenfolge

1. P1 Host: `admin/users` Create, `admin/users` Detail, `admin/groups` Detail, `admin/groups` Membership, `admin/organizations` Membership, `admin/instances` Create, `admin/instances` Detail-Config, `admin/legal-texts` Create, `admin/legal-texts` Detail, `interfaces` Mainserver-Form, `interfaces` S3-Form, `interfaces` Supabase-Form, `content`.
2. P1 Plugins: `plugin-poi`-Editor, Waste `master-data` Fraction-Dialog, Waste `master-data` Location-Form, Waste `tours`-Form.
3. P2 Host: `admin/groups` Create, `admin/organizations` Create + Detail-Stammdaten, `admin/roles` Create + General-Tab, `interfaces` Type-Picker.
4. P2 Plugins: Waste `master-data` Region-/City-/Street-/HouseNumber-Dialoge, Waste `settings`, Waste `scheduling` global + tour.
5. P3 Sonderfall: `admin/roles` Rechtekatalog nur teilweise auf RHF heben, weil Tabellen-Bulk-Editor kein klassisches Feldformular ist.

## Auffälligkeiten für die Migration

- Mehrere Flows validieren erst im Submit oder erst im Mapper. Das betrifft besonders `interfaces`, Waste `master-data`, Waste `scheduling`, Waste `tours` und Teile von `admin/users`.
- Datumslogik mit DST-/Europe-Berlin-Bezug liegt derzeit in manuellen Parsern (`groups`, `legal-texts`, `content`) und sollte in wiederverwendbare RHF-Resolver-Regeln überführt werden.
- Einige Flows führen pro Submit mehrere Mutationen aus (`groups` Detail, `plugin-poi` mit Host-Media, `instances` Create + Bootstrap als Folgeaktion). Diese Pfade brauchen für die Migration priorisiert MSW-basierte HTTP-Tests statt reinem Hook-Mocking.
- Gute fast-check-Kandidaten bleiben die konkret benannten Mapper/Validatoren/Sortierfunktionen je Zeile; visuelle Dialoghüllen ohne eigene Logik sind keine primären Kandidaten.
